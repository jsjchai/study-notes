## 1. 问题：OpenClaw Gateway CPU 100% 卡顿

### 问题现象

2026-03-30 下午，OpenClaw Gateway 主进程（`openclaw-gateway`，PID 15057）CPU 占用持续 **100%**，内存约 1 GB，持续时间超过 15 分钟。Gateway 响应极慢，所有 WebSocket 请求延迟剧增。

### 影响范围

- webchat UI 完全不可用
- 所有 Cron 任务执行超时
- Gateway 日志显示 `sessions.list` 调用每次耗时 **5+ 秒**
- 14:00-17:00 时段 Cron 任务大量失败（Rate Limit 触发）

每个 `sessions.list` 调用耗时 **5+ 秒**，且持续不断。

### 第三步：分析 sessions.json 体积

```bash
$ wc -c ~/.openclaw/agents/main/sessions/sessions.json
4203269  sessions.json   # 4.2 MB
```

`sessions.json` 文件达到 **4.2 MB**，这是问题根源。每次 `sessions.list` 调用都要解析整个 JSON 文件。

### 第四步：检查 session 数量和内容

```bash
$ openclaw sessions --all-agents --json | python3 -c "import json,sys; d=json.load(sys.stdin); print(f'总数: {d[\"count\"]}')"
总数: 293
```

293 个 session 条目，远超正常数量。

### 第五步：分析 session 堆积原因

```python
# 分析 skillsSnapshot 体积
for k, v in data.items():
    snap = v.get('skillsSnapshot', {})
    size = len(json.dumps(snap))
    print(f'{k}: skillsSnapshot size={size}')
```

| session 类型 | 示例 | 数量 |
|------------|------|------|
| cron 主任务 | `cron:76103608-...` | ~9 个 |
| cron 执行残留 | `cron:...:run:xxx` | 大量（每次执行留下） |
| subagent | `agent:main:subagent:uuid` | ~200+ 个 |
| main | `agent:main:main` | 1 个 |

每个 entry 内嵌完整 `skillsSnapshot`（~10 KB/个），是文件膨胀的主因。

### 第六步：定位根本原因——session.maintenance 默认配置

```bash
$ openclaw config get session.maintenance
# 无输出（未配置）
```

**关键发现：** `session.maintenance.mode` 默认值是 `"warn"`（只报告不清理）。这意味着：
- 293 个 session 条目全部被保留
- Cron/subagent session 执行完成后从不自动清理
- sessions.json 只增不减

### 第七步：OpenClaw Session 创建逻辑分析

根据 `concepts/session.md`，session 创建时机如下：

| 来源 | Session Key | 说明 |
|------|-----------|------|
| WebSocket 连接 | `agent:<id>:<dmScope>` | 每次连接创建 |
| Cron 主任务 | `cron:<jobId>` | Cron 调度时创建 |
| Cron 子执行 | `cron:<jobId>:run:<runId>` | **每次执行都创建新 session** |
| Subagent spawn | `agent:main:subagent:<uuid>` | 每次 spawn 创建 |
| Webhook | `hook:<uuid>` | 每次触发创建 |

> **核心问题：** `sessions.json` 本身是 append-only 的注册表，只有 session key 元数据（不含 transcript），但大量累积后每次 list 调用都要解析全量 JSON。

## 3. 解决办法

### 方案一：立即清理（临时修复）

**手动清理超过 24 小时的 subagent/cron session 条目：**

```python
import json
import os
import time

sessions_file = os.path.expanduser("~/.openclaw/agents/main/sessions/sessions.json")
cutoff_ms = 24 * 3600 * 1000  # 24小时

with open(sessions_file, 'r') as f:
    data = json.load(f)

now_ms = int(time.time() * 1000)
cleaned = {
    k: v for k, v in data.items()
    if k == "agent:main:main" or (now_ms - v.get("updatedAt", 0) < cutoff_ms)
}

with open(sessions_file, 'w') as f:
    json.dump(cleaned, f, indent=2)

print(f"清理: {len(data)} → {len(cleaned)} entries")
```

**结果：**
```
清理前: 292 entries
清理后: 80 entries
sessions.json: 4.2 MB → 1.0 MB
```

### 方案二：配置 session.maintenance（根本解决）

```bash
$ openclaw config set session.maintenance.mode enforce
$ openclaw config set session.maintenance.pruneAfter 7d
$ openclaw config set session.maintenance.maxEntries 100
$ openclaw config set session.maintenance.rotateBytes 5mb
$ openclaw gateway restart
```

**配置含义：**

| 配置项 | 值 | 说明 |
|--------|-----|------|
| `mode` | `enforce` | 自动执行清理（默认 warn 只报告）|
| `pruneAfter` | `7d` | 超过7天的 session 标记为 stale 并清理 |
| `maxEntries` | `100` | 最多保留 100 个 session 条目 |
| `rotateBytes` | `5mb` | sessions.json 超过 5MB 时轮转 |

**enforce 模式清理顺序：**
1. 删除超过 `pruneAfter` 的 stale 条目
2. 按 `maxEntries` 裁剪最旧的条目
3. 归档不再被引用的 transcript 文件
4. 清理过期归档文件
5. sessions.json 超过 `rotateBytes` 时轮转

**配置验证：**
```bash
$ openclaw config get session.maintenance
{
  "mode": "enforce",
  "pruneAfter": "7d",
  "maxEntries": 100,
  "rotateBytes": "5mb"
}
```

### 方案三：新增每日自动清理 Cron（兜底）

即使配置了 maintenance，也可以添加每日 Cron 作为双保险：

```json
{
  "name": "openclaw-session-cleanup",
  "schedule": "0 6 * * *",
  "payload": {
    "kind": "agentTurn",
    "message": "执行 session 清理：\n1. openclaw sessions cleanup --all-agents --enforce\n2. 检查 session 数量，超100则手动清理"
  }
}
```

## 4. 总结

| 维度 | 优化前 | 优化后 |
|------|--------|--------|
| sessions.json 大小 | 4.2 MB | **38 KB** |
| session 条目数 | 293 | ~10（自动清理生效）|
| sessions.list 耗时 | **5+ 秒** | **毫秒级** |
| maintenance.mode | `warn`（不清理）| `enforce`（自动清理）|
| 保留时间 | 永久 | **7 天** |
| 数量上限 | 无 | **100 条** |

**核心教训：**
1. `session.maintenance.mode` 默认 `warn`，不会自动清理任何 session
2. Cron/subagent 每次执行都创建新 session key，如不及时清理会快速膨胀
3. skillsSnapshot 内嵌在每个 session entry 中（~10 KB/个），大量累积后 JSON 解析变慢
4. 建议在部署时直接配置 `session.maintenance.mode=enforce`，防止 session 无限增长
