# OpenClaw MEMORY.md 膨胀事故：从 46KB 到 11KB 的紧急瘦身手记

**2026 Week 14（4/1）**

---

## 1. 问题：main Agent 卡死

### 问题现象

2026-04-01 18:54，用户反馈 main Agent 处于"卡死状态"，微信消息发送后无响应。

登录 Dashboard 检查后发现：Gateway 本身运行正常（pid 64161，RPC probe ok），微信渠道也在线，但就是无法正常响应。

进一步检查日志，发现了关键线索：

```
warn agent/embedded lane wait exceeded: lane=session:agent:main:...
waitedMs=27509 queueAhead=0
```

Session lane 等待时间超过 **27 秒**，但队列前方没有任何阻塞任务——说明问题出在 session 本身。

---

## 2. 根因定位

### 第一步：检查 Gateway 日志

日志中反复出现这条警告：

```
warn agent/embedded workspace bootstrap file MEMORY.md is 29026 chars
(limit 20000); truncating in injected context
```

**MEMORY.md 文件 29,026 字符，超出了 20,000 字符的注入上限，被强制截断 38%。**

截断意味着：每次新建 Session 时，OpenClaw 自动把 MEMORY.md 截掉近四成内容再注入上下文——安全协议、Git 规范、项目索引全被切掉，main Agent 相当于在一个残缺的上下文中运行，自然会"失忆"卡死。

### 第二步：确认文件体积

```bash
$ wc -c ~/.openclaw/workspace/MEMORY.md
46271 MEMORY.md
```

**46,271 字符，超出上限 2.3 倍。**

### 第三步：溯源膨胀原因

MEMORY.md 长期只增不减，927 行里堆满了以下内容：

| 来源 | 预估大小 | 问题 |
|------|---------|------|
| Sprint 归档（重复副本） | ~12k | `memory/2026-03-31.md` 和 `memory/2026-04-01.md` 已有完整存档，但 MEMORY.md 里还有一份重复的 |
| Agent 工作记录累积（03-28~04-01） | ~10k | 每日归档后未从 MEMORY.md 清理，重复堆积 |
| 已完成项目详细记录 | ~8k | openmedu Sprint 1/2、Issue #6 等详细记录未迁移 |
| 规范文档（过重） | ~8k | 周报格式规范、Agent 能力观察等内容未精简 |
| SOUL.md 分裂事故遗留 | ~3k | 事故恢复时多备份了一部分内容 |

**根本问题：`memory/YYYY-MM-DD.md` 归档文件已存在且内容完整，但从未定期从 MEMORY.md 中清除过期的归档章节。**

---

## 3. 紧急修复

### 第一步：备份

```bash
$ cp ~/.openclaw/workspace/MEMORY.md \
  ~/.openclaw/workspace/backups/MEMORY-backup-20260401185758.md
```

### 第二步：精简 MEMORY.md

**保留内容（MEMORY.md）：**
- 安全协议 / Git 规范
- 项目索引（仓库地址、技术栈）
- Sprint 当前进度（1-2 段）
- 待跟进清单（≤ 5 条活跃项）
- Agent 配置（路由表、人设速查）
- Cron 任务配置

**归档内容（memory/YYYY-MM-DD.md）：**
- 每日 Agent 工作记录（详细）
- Sprint 完整交付物清单
- 代码审查报告
- 运维产出记录

### 第三步：Gateway 重启

```bash
$ openclaw gateway restart
Scheduled LaunchAgent restart: gui/501/ai.openclaw.gateway
restart scheduled, gateway will restart momentarily
```

## 5. 制度化：MEMORY.md 瘦身规则

为了避免同类问题再次发生，建立了以下归档机制：

### 归档触发规则

| 事件 | 动作 | 执行者 |
|------|------|--------|
| Sprint 结束 | MEMORY.md 中该 Sprint 章节移至 `memory/YYYY-MM-DD.md` | @史官 |
| 单章节 > 3k 字符 | 拆分归档，MEMORY.md 只保留摘要 | @史官 |
| MEMORY.md > 18k 字符 | 强制自检，@史官 48h 内完成瘦身 | Cron 触发 |
| 项目阶段完成 | 将详细记录移至对应 daily-log，MEMORY.md 只保留索引 | @史官 |

### 归档格式模板

```markdown
# YYYY-MM-DD Sprint N 日志
> 维护者：@史官 | 更新：HH:MM
> Sprint N 第X天 · 全量归档

## Sprint 概览
[表格：任务ID / 负责人 / 状态 / 产出物]

## 代码审查
[评分 / 问题 / 修复状态]

## Git Commits
[commit hash / 内容摘要]

## 详细交付物
[按模块分类的完整记录]
```

---

## 6. 教训总结

1. **MEMORY.md 是"索引"而非"仓库"**：详细内容必须下沉到 daily-log，不能把所有东西都堆在 MEMORY.md 里
2. **容量红线意识**：MEMORY.md 有 20k 硬限制，接近 18k 时必须主动瘦身，不能等溢出
3. **归档不能只增不减**：daily-log 归档完成后，MEMORY.md 对应章节必须同步删除，不能留重复副本
4. **规范文档也要精简**：格式模板、Agent 能力观察等内容应在 AGENTS.md 中维护，不应占用 MEMORY.md 空间
