## 根因分析

**直接原因：** `MEMORY.md` 文件超限（**46k 字符 / 20k 限制**），每次新建 Session 时 OpenClaw 自动截断 38% 上下文，导致 main Agent 丢失关键记忆进入"卡死"状态。

**深层原因：长期只增不减，缺乏归档机制**

### 膨胀来源拆解（927行 / 46k）

| 来源 | 字符数 | 问题 |
|------|--------|------|
| 历史 Sprint 归档（重复） | ~12k | `memory/2026-03-31.md` 和 `memory/2026-04-01.md` 已有完整存档，MEMORY.md 里是重复副本 |
| Agent 工作记录累积（28-31日） | ~10k | 每日归档未从 MEMORY.md 清理，重复堆积 |
| 已完成项目详细记录（过细） | ~8k | openmedu Sprint 1/2、Issue #6 等历史详细记录未迁移 |
| 规范文档（过重） | ~8k | 周报格式规范（4000字符）、Agent 能力观察（3000字符）等未精简 |
| SOUL.md 分裂事故遗留 | ~3k | 事故恢复时可能多备份了一部分内容 |

**根本问题：** `memory/YYYY-MM-DD.md` 归档文件已存在且内容完整，但从未定期从 MEMORY.md 清除过期的归档章节。

## 修复方案

### 1. 紧急修复（当日执行）

```bash
# 备份
cp MEMORY.md backups/MEMORY-backup-20260401185758.md

# 瘦身
# 保留：安全协议 / Git规范 / 项目索引 / Sprint进度 / 待跟进≤5条 / Agent配置
# 归档：Sprint详细记录 / Agent工作日志 → memory/YYYY-MM-DD.md
# 清理：重复归档 / 过细项目记录 / 过重规范章节
```

### 2. MEMORY.md 瘦身规则（制度化）

**目标：** MEMORY.md 始终 ≤ 20k 字符

#### 保留内容（MEMORY.md）
- 安全协议 / Git 规范
- 项目索引（仓库地址、技术栈）
- Sprint 当前进度（1-2 段）
- 待跟进清单（≤ 5 条活跃项）
- Agent 配置（路由表、人设速查）
- Cron 任务配置

#### 归档内容（memory/YYYY-MM-DD.md）
- 每日 Agent 工作记录（详细）
- Sprint 完整交付物清单
- 代码审查报告
- 运维产出记录
- 项目阶段总结（脚手架/POC/架构决策）

#### 归档触发规则

| 事件 | 动作 | 执行者 |
|------|------|--------|
| Sprint 结束 | MEMORY.md 中该 Sprint 章节移至 `memory/YYYY-MM-DD.md` | @史官 |
| 单章节 > 3k 字符 | 拆分归档，MEMORY.md 只保留摘要 | @史官 |
| MEMORY.md > 18k 字符 | 强制自检，@史官 48h 内完成瘦身 | Cron 触发 |
| 项目阶段完成 | 将详细记录移至对应 daily-log，MEMORY.md 只保留索引 | @史官 |

#### 归档格式模板

```markdown
# YYYY-MM-DD Sprint N 日志
> 维护者：@史官 | 更新：HH:MM
> Sprint N 第X天 · 全量归档

## Sprint 概览
[表格：任务ID / 负责人 / 状态 / 产出物]

## 代码审查
[评分 / 问题 / 修复状态]

## Git Commits
[commit hash / 内容摘要]

## 详细交付物
[按模块分类的完整记录]
```

## 教训总结

1. **归档不能只增不减**：MEMORY.md 是"索引"而非"仓库"，详细内容必须下沉到 daily-log
2. **容量红线意识**：MEMORY.md 有 20k 硬限制，接近 18k 时必须主动瘦身，不能等溢出
3. **规范文档也要精简**：格式模板、Agent 能力观察等内容应在 AGENTS.md 中维护，不应占用 MEMORY.md 空间
4. **daily-log 是第一公民**：每日归档完成后，MEMORY.md 对应章节必须同步删除，不能留重复副本
